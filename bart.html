BART and BART for Causal Inference

This section introduces the concept of regression trees, Bayesian inference, and BART. Then extends to BART for Causal Inference.

Our Data Example
Let's consider a hypothetical example where you were given data from a recent running race with 200 runners. Some of the runners wore the newly developed HyperShoes, while some of the runners wore standard shoes. The treatment (wearing HyperShoes) was not randomly assigned, runners choose for themselves which shoes to use.

Besides the treatment variable Z (HyperShoes or standard shoes) and outcome variable Y(running times), the dataset has measurements of age.

How can we determine if wearing HyperShoes caused runners to run faster than they would have run had they worn standard shoes?

Regression Trees
We're going to temporarily ignore the causal question for the sake of understanding regression trees.
Regression trees are simple models that partition the data by finding the best splits. A split is a rule based on one of the covariates (features), such as “is age < 40?” or “is Z = 1?”. For example, a model like rpart(running_time ~ z + age) uses the variables z (treatment) and age to explain variation in running_time.

Rather than fitting a single equation, a regression tree repeatedly splits the data into smaller groups. Within each group, it predicts the outcome using the average (mean) of the observed values in that group.

Looking at the process step by step:
Initial state: The model starts with points forming two rough lines. The available covariates are z (treatment) and age.

First split: The model chooses the split that best reduces prediction error. In this case, it splits on treatment (Z = 1 vs. Z = 0), dividing the data into two groups. We show this by color now being added to the graph. Each group is then assigned a predicted value equal to its mean running time.

Further splits: One split is usually not enough to capture the structure in the data. The model continues splitting. For example, now on age within each treatment group—creating smaller regions where the mean better approximates the data.

Stopping: This process continues until a stopping rule is met. "Enough" can be controlled by parameters we set before running the model.

Be warned! Regression trees are very prone to overfitting, as seen in the later images. Those extra splits may improve fit on the data but usually harm performance on data we haven't seen yet. At the same time, we do want enough flexibility to capture real nonlinear relationships. This creates a tradeoff: too few splits underfits, too many splits overfit.

So we need something that can more automatically regulate how many splits a regression tree is allowed to make given the data.

Additive Regression Trees
For a more flexible model (that fits non-linearities well) but also guards against overfitting, one improvement is Gradient Boosted Regression Trees. These models build many small regression trees.

The idea is to add the contributions of each weak tree together, as a tree is 'boosted' by previous ones.

The newly generalized trees are kept weak by setting the depth (number of splits) of the weak trees, number of weak trees, and with a shrinkage parameter that is multiplied to the fit of each tree (called a learning rate). Each of these three parameters are tunable and set with cross validation.

How it works:
1. Use the mean outcome as the prediction
2. Fit a small tree to improve our prediction
3. Look at the residuals from the model (what it could not explain)
4. Fit another small tree on those residuals, multiplied by the learning rate (this slows down how fast the generalized tree is learning)
5. Repeat (2-4), usually 75 - 150 times. This ends up looking like our previous single regression tree animation, but is different in methodology (for the better!).

And this creates a much better fit to our data from the initial single regression tree.

The issue is that this method depends a lot on cross validation that takes a lot of time, and left with single predictions without telling us how confident they were in that prediction. So how can we do even better? Yes - with BART.

Bayesian Inference
BART trades traditional hyperparameters found with cross-validation for Bayesian priors. Bayesian priors can be interpreted here as 'what are our prior beliefs about regression trees?'

1. We should be skeptical about predictions far away from the general mean. In an example with which we had no other information, our best guess is the mean. So we should be wary of extreme things as a sign of overfit.
2. The weak trees should be pretty small (kept weak).
3. Our model should be able to outperform a linear regression. With no traditional linear regression parametric assumptions, we can do better than a straight line.

These verbal priors have to then be formed mechanically. With BART, they're calibrated from the data (meaning that its performs well at default).
Priors, by importance:

1. Prior on the Means. We want our predictions to be generally closer to the mean, so extreme predictions get pulled towards the mean.

2 Prior on the Tree Depth. When a tree is proposed, it has a certain depth (number of cuts), and we believe we want more depth=2 or depth=3 than depth = 1 (no cuts) or depth = 4,5. This isn't a hard cut-off, it just needs to see a strong signal to delve deeper (which is nice about a Bayesian prior).

3. Prior on the Standard Deviation. We'll fit a linear regression and look at its standard deviation. This prior pulls things such that we have a smaller standard deviation than linear regression (we should be doing better than it, after all).

Trees are proposed (tried out) and either accepted or rejected with a Metropolis-Hastings algorithm (based on gradual improvement).

BART
This is where we 'put it all together'.

Running a BART Model first scales the outcome, proposes a variety of general trees:

1. What if we grow a new weak tree
2. What if we prune an existing tree
3. What if we flip an existing tree
4. What if we add a new split on an existing tree
Then BART accepts or rejects trees, such that we gradually improve performance. And repeats over many (thousands) of iterations!

Using BART

For recreating plots or playing with your own data for predictive reasons, we reccomend the dbarts R library with the bart2 function.
e.g.) fit <- bart2(y ~ covariate + covariate,data=runners)

For unseen data, we reccomend using the posterior predictive distribution as it makes our estimates more uncertain.

Partial Pooling

Sometimes we want to use more sophisticated models. Say that we also, in our data, tracked which country these runners came from.

We can extend the BART framework to accomodate for countries in new data from countries we haven't seen before. Traditional models are stuck because if we modelled country as a fixed factor, and got a person from a new country, the traditional model doesn't know how to account for that.

The stan4bart package in R is best for "Partial Pooling", where as in this example it would treat each country as having its own 'population'. Here, stan4bart gets around that 'stuckness' by modeling the variance across countries than each countries means. That way, when a new country appears that stan4bart needs to predict, it doesn't change anything about implementation. (i.e., it does not need to be seen in the training data)

Why use Partial Pooling?
* accounts for unseen future groups
* non-independent and identically distributed observations
* more regularization
* improved predictions assuming independence
* more efficient assuming independence

(link to stan4bart paper) Since this has been so far a predictive problem, we've treated z like a covariate and only have been interested in predicting data well.

BART for Causal Inference
Let's return to a causal framework. Recall that we wanted to determine if wearing HyperShoes caused runners to run faster than they would have run had they worn standard shoes.
We'll also expand our data to a larger age range, but we only observe people over 50 without the hyperShoe. We'll also make new data points so this feels a little different than the predictive section, but looks the same.

Choosing an Estimand

In this example, we’ll be estimating the effect of the treatment on the treated, or ATT. It appears that every runner under and up to 50 wore HyperShoes. Without runners older than 50 in the treatment group, we don’t have enough available information to infer about counterfactuals for the average treatment effect (ATE) or the effect of the treatment on the controls (ATC) but we do have complete overlap for the ATT, and we have enough information to infer about what would have happened if those participants had not worn HyperShoes!

In a completely randomized experiment (where everyone has the same probability of receiving the treatment) the ATE, ATT and ATC are expected to be the same because there are no systematic differences between the treatment and control groups.
So whether we average individual level treatment effects over the treatment group or the control group or the whole sample shouldn't make a difference (in expectation). This is not the case for observational studies where the ATE, ATT and ATC can be different. This is because in observational studies, the participants in the treatment group and participants in the control group can look very different from one another.

Imbalance
In observational studies, the treatment is not randomly assigned and we no longer expect covariates to be balanced between the treatment and control group. So it's not surprising in this study that on average, runners in the treatment group are about 14 years younger than the runners in the control group.

We will need to rely on a statistical model to correct this imbalance or else the substantial difference in age will bias our estimate of the ATT.

Suspending Disbelief to View Potential Outcomes
Before we begin analyzing our observational data, let's suspend belief and see what our results would look like if we had access to each runner's factual and counterfactual outcomes. With all potential outcomes, we can calculate each runner's true individual causal effect (ICE). In this plot we also can see that age is linearly related to running times. Each runner's ICE is shown as a black line. Keep in mind that because we are interested in the ATT, we only need to predict the counterfactuals of runners in the treatment group!

Unfortunately, in practice we never have access to all potential outcomes and can never make this calculation. But in order to understand the properties of our estimators we need to be able to understand how they will perform in circumstances where we know the truth and the assumptions are satisfied. When we suspend disbelief and have access to all potential outcomes, we can calculate the true ATT of our sample by taking the average of all the individual causal effects for runners who received the treatment (wore HyperShoes) in the real world.

Calculating True ATT
When we average the individual causal effects of all the runners in the treatment group, we can see that the true ATT from our sample is -55.16. This is the true ATT that we will be attempting to estimate. Now we will return to reality where we only have access to our observed data and need to estimate the ATT.



Difference in Means
If we use the difference in means as our statistical model, our estimate would be -12.6 which is extremely far from true ATT! The story is the same as the last time we used comparing the difference in means to estimate the ATT, we are still not incorporating any information about age and the imbalance of age has not been accounted for. Not appropriately adjusting for the confounding variable of age explains why the estimated ATT is so biased.



Linear Regression
When the relationships between covariates and the outcome are non-linear, the linear regression estimate for the ATT is likely to be biased.
In our situation the estimate is -51.8 which is noticeably different from the true ATT! While linear regression attempts to account for the imbalance of age, the model simply doesn't fit the data well and thus can't provide a good prediction of the counterfactual state!

Linear regression assumes that the relationships between predictors and the response are linear. When the treatment is not randomly assigned we can no longer expect balance between groups and need to rely more on our statistical model. In the case of linear regression, this means leaning on the assumption that relationships are linear which may or may not be true! When this assumption is not true, estimates from a linear regression are likely to be biased. This assumption is less of a problem when the treatment is randomly assigned and we'll have a new module coming soon that explains why.

In our current observational study, it is tempting to think about transforming the data, adding polynomial terms or adding interactions to the regression to improve the fit. While this might work once in a while when only a few predictors are involved, in practice this is not a sustainable solution. Imagine if we had 10, 20 or even 200 confounders we needed to account for! Identifying which variables need to be transformed or which interaction term would need to be added would quickly become impossible.
