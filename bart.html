BART and BART for Causal Inference

This section introduces the concept of regression trees, Bayesian inference, and BART. Then extends to BART for Causal Inference.

Our Data Example
Let's consider a hypothetical example where you were given data from a recent running race with 200 runners. Some of the runners wore the newly developed HyperShoes, while some of the runners wore standard shoes. The treatment (wearing HyperShoes) was not randomly assigned, runners choose for themselves which shoes to use.

Besides the treatment variable Z (HyperShoes or standard shoes) and outcome variable Y(running times), the dataset has measurements of age.

How can we determine if wearing HyperShoes caused runners to run faster than they would have run had they worn standard shoes?

Regression Trees
We're going to temporarily ignore the causal question for the sake of understanding regression trees.
Regression trees are simple models that partition the data by finding the best splits. A split is a rule based on one of the covariates (features), such as “is age < 40?” or “is Z = 1?”. For example, a model like rpart(running_time ~ z + age) uses the variables z (treatment) and age to explain variation in running_time.

Rather than fitting a single equation, a regression tree repeatedly splits the data into smaller groups. Within each group, it predicts the outcome using the average (mean) of the observed values in that group.

Looking at the process step by step:
Initial state: The model starts with points forming two rough lines. The available covariates are z (treatment) and age.

First split: The model chooses the split that best reduces prediction error. In this case, it splits on treatment (Z = 1 vs. Z = 0), dividing the data into two groups. We show this by color now being added to the graph. Each group is then assigned a predicted value equal to its mean running time.

Further splits: One split is usually not enough to capture the structure in the data. The model continues splitting. For example, now on age within each treatment group—creating smaller regions where the mean better approximates the data.

Stopping: This process continues until a stopping rule is met. "Enough" can be controlled by parameters we set before running the model.

Be warned! Regression trees are very prone to overfitting, as seen in the later images. Those extra splits may improve fit on the data but usually harm performance on data we haven't seen yet. At the same time, we do want enough flexibility to capture real nonlinear relationships. This creates a tradeoff: too few splits underfits, too many splits overfit.

So we need something that can more automatically regulate how many splits a regression tree is allowed to make given the data.

Additive Regression Trees
For a more flexible model (that fits non-linearities well) but also guards against overfitting, one improvement is Gradient Boosted Regression Trees. These models build many small regression trees.

The idea is to add the contributions of each weak tree together, as a tree is 'boosted' by previous ones.

The newly generalized trees are kept weak by setting the depth (number of splits) of the weak trees, number of weak trees, and with a shrinkage parameter that is multiplied to the fit of each tree (called a learning rate). Each of these three parameters are tunable and set with cross validation.

How it works:
1. Use the mean outcome as the prediction
2. Fit a small tree to improve our prediction
3. Look at the residuals from the model (what it could not explain)
4. Fit another small tree on those residuals, multiplied by the learning rate (this slows down how fast the generalized tree is learning)
5. Repeat (2-4), usually 75 - 150 times. This ends up looking like our previous single regression tree animation, but is different in methodology (for the better!).

And this creates a much better fit to our data from the initial single regression tree.

The issue is that this method depends a lot on cross validation that takes a lot of time, and left with single predictions without telling us how confident they were in that prediction. So how can we do even better? Yes - with BART.

Bayesian Inference
BART trades traditional hyperparameters found with cross-validation for Bayesian priors. Bayesian priors can be interpreted here as 'what are our prior beliefs about regression trees?'

1. We should be skeptical about predictions far away from the general mean. In an example with which we had no other information, our best guess is the mean. So we should be wary of extreme things as a sign of overfit.
2. The weak trees should be pretty small (kept weak).
3. Our model should be able to outperform a linear regression. With no traditional linear regression parametric assumptions, we can do better than a straight line.

These verbal priors have to then be formed mechanically. With BART, they're calibrated from the data (meaning that its performs well at default).
Priors, by importance:

1. Prior on the Means. We want our predictions to be generally closer to the mean, so extreme predictions get pulled towards the mean.

2 Prior on the Tree Depth. When a tree is proposed, it has a certain depth (number of cuts), and we believe we want more depth=2 or depth=3 than depth = 1 (no cuts) or depth = 4,5. This isn't a hard cut-off, it just needs to see a strong signal to delve deeper (which is nice about a Bayesian prior).

3. Prior on the Standard Deviation. We'll fit a linear regression and look at its standard deviation. This prior pulls things such that we have a smaller standard deviation than linear regression (we should be doing better than it, after all).

Trees are proposed (tried out) and either accepted or rejected with a Metropolis-Hastings algorithm (based on gradual improvement).

BART
This is where we 'put it all together'.

Running a BART Model first scales the outcome, proposes a variety of general trees:

1. What if we grow a new weak tree
2. What if we prune an existing tree
3. What if we flip an existing tree
4. What if we add a new split on an existing tree
Then BART accepts or rejects trees, such that we gradually improve performance. And repeats over many (thousands) of iterations!
